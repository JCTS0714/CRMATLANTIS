# Progreso del Desarrollo - 5 de Marzo 2026

## Resumen Ejecutivo

Durante esta sesion se implemento el modulo **Scrum > Tareas** de extremo a extremo:

- Interfaz nueva en sidebar y pantalla dedicada.
- Flujo completo en frontend (kanban + lista + modal de creacion/edicion).
- Persistencia real en base de datos (sin datos ficticios).
- API REST para listar, crear, editar, cambiar estado y buscar responsables.
- Reglas finales solicitadas:
  - Todos los usuarios autenticados pueden ver y editar tareas.
  - `asignador` se autocompleta con el usuario logueado y no se puede editar.
  - Estado solo por drag and drop en kanban.
  - Campo funcional de estado de tiempo (`estado_tiempo`: `0`/`1`) mostrado como `A tiempo` / `Retrasado`.

## Alcance Implementado

### 1. Navegacion y estructura de interfaz

Se agrego un nuevo bloque en el sidebar:

- Menu: `Scrum`
- Submenu: `Tareas`
- Ruta visual: `/scrum/tareas`

Tambien se integro en la cabecera principal de `App.vue`:

- Boton `Nueva tarea`.
- Toggle de vista `kanban <-> lista` via querystring (`?view=kanban|list`).

### 2. Vista Scrum Tareas (frontend)

Se construyo el componente `ScrumTasksView.vue` con:

- Filtros por:
  - texto
  - responsable
  - prioridad
- Vista kanban con columnas:
  - `pendiente`
  - `en_progreso`
  - `completada`
- Vista lista con columnas:
  - Nombre
  - Asignador
  - Responsable
  - Prioridad
  - Ejecucion
  - Estado tiempo
  - Estado
  - Acciones

### 3. Flujo de edicion y estado

Cambios de UX aplicados segun requerimientos:

- Se elimino el selector de estado dentro del modal.
- Se elimino la accion de cambio de estado en tarjeta fuera de drag and drop.
- La tarjeta en kanban ahora se edita con click.
- El estado se cambia solo arrastrando entre columnas.

### 4. Formulario de tarea

Campos implementados:

- `nombre`
- `descripcion`
- `asignador`
- `responsable`
- `prioridad`
- `tiempo_ejecucion`
- `observacion`

Comportamiento aplicado:

- `asignador`:
  - Se llena con el usuario autenticado.
  - Queda en solo lectura en el frontend.
  - Se protege en backend para impedir cambios manuales.
- `responsable`:
  - Buscador remoto de usuarios.
  - Permite reasignar.
  - Sin exclusion del usuario logueado (segun solicitud final).
- `prioridad`:
  - Precarga `tiempo_ejecucion` por defecto:
    - alta: +1 dia
    - media: +2 dias
    - baja: +3 dias
  - Fecha editable por usuario.

### 1. Migracion

Se creo la tabla `scrum_tasks` con:

- `nombre`
- `descripcion`

- `timestamps`

### 2. Modelo

Se agrego `App\Models\ScrumTask` con:

- `fillable` para campos de tarea.
- cast `tiempo_ejecucion` como `datetime`.
- relaciones:
  - `asignador()`
  - `responsable()`

### 3. Controlador

Se agrego `App\Http\Controllers\Scrum\ScrumTaskController` con endpoints para:

- listar tareas
- crear tarea
- actualizar tarea
- cambiar estado
- buscar responsables

Validaciones y reglas clave:

- `asignador_id` y `asignador` quedan `prohibited` en `store` y `update`.
- El backend fija `asignador_id` con `request()->user()->id` al crear.
- Sin restricciones adicionales por usuario para ver/editar tareas.

## Estado de Tiempo (`estado_tiempo`)

Se formalizo el estado de tiempo como campo de salida en API:

- `0` = `A tiempo`
- `1` = `Retrasado`

Regla aplicada en backend (helper `resolveEstadoTiempo`):

- Si no hay `tiempo_ejecucion`: `0`
- Si la tarea esta `completada`: `0`
- Si vencio y no esta completada: `1`
- Si aun no vence: `0`

Representacion en UI:

- Badge en tarjetas kanban.
- Columna `Estado tiempo` en vista lista (ubicada despues de `Ejecucion`, como se solicito).
