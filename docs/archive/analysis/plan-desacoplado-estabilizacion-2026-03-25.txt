PLAN DESACOPLADO DE ESTABILIZACION
Proyecto: CRM ATLANTIS
Fecha: 2026-03-25

OBJETIVO
Ejecutar mejoras en bloques independientes para evitar que arreglar una parte rompa otra.
Cada paso debe poder implementarse, validarse y revertirse por separado.

PRINCIPIOS DE EJECUCION
1) Independencia por dominio
- Cada paso toca un solo dominio tecnico/funcional.
- No mezclar en un mismo cambio: permisos + UI, rutas + negocio, scheduler + frontend.

2) Contrato estable
- Antes de cada cambio, definir que no cambia:
  - rutas publicas
  - payloads JSON
  - nombres de permisos/roles (si no es una fase RBAC)
- Si se requiere romper contrato, abrir fase separada.

3) Validacion con corte
- Cada paso termina con:
  - smoke test del dominio
  - checklist funcional minima
  - criterio de rollback explicito

4) Reversibilidad
- Si falla, se revierte TODO el paso.
- Prohibido dejar cambios parciales para el siguiente paso.

REGLAS DE CONTROL (ANTI-REGRESION)
1) Un PR = un dominio.
2) Maximo 8-12 archivos por PR (salvo excepcion justificada).
3) Prohibido tocar RBAC y navegacion en el mismo PR.
4) Prohibido tocar rutas y logica de negocio en el mismo PR.
5) Cada PR incluye:
- alcance
- no-alcance
- riesgos
- plan de rollback
- evidencia de pruebas

SECUENCIA PROPUESTA (DESACOPLADA)
PASO 1: Observabilidad y metricas
- Alcance: logs, tiempos de respuesta, contadores de uso por modulo.
- No cambia funcionalidad.
- Objetivo: medir antes de optimizar.

PASO 2: Carga recurrente de frontend
- Alcance: polling de notificaciones/dashboard (intervalos, estrategia adaptativa).
- Sin cambios en RBAC, rutas o permisos.
- Objetivo: reducir carga por usuario concurrente.

PASO 3: Scheduler y colas
- Alcance: ajuste de tareas programadas y delegacion a jobs en cola.
- Sin cambios de UI.
- Objetivo: desacoplar procesos pesados del request lifecycle.

PASO 4: RBAC interno (roles/permisos)
- Alcance: simplificar fuentes de verdad de permisos, eliminar duplicidad de middlewares equivalentes.
- Sin tocar layout/menu visual.
- Objetivo: bajar complejidad de mantenimiento de seguridad.

PASO 5: Navegacion y visibilidad de menu
- Alcance: visibilidad en sidebar y catalogo de menu.*
- Sin cambiar autorizacion backend.
- Objetivo: coherencia UI sin riesgo de seguridad.

PASO 6: Modularizacion de rutas
- Alcance: dividir archivo de rutas por dominio.
- Mantener mismos nombres de ruta y middlewares.
- Objetivo: reducir friccion de mantenimiento.

PASO 7: Refactor frontend estructural
- Alcance: extraer subcomponentes/composables de archivos grandes.
- Sin tocar APIs.
- Objetivo: reducir deuda en componentes monoliticos.

PASO 8: Pruebas y compatibilidad de migraciones
- Alcance: estabilizar suite de tests y ajustes cross-db en migraciones.
- Sin agregar features.
- Objetivo: liberar cambios con confianza.

PASO 9: Poda funcional
- Alcance: desactivar o retirar funciones de bajo uso con deprecacion controlada.
- Sin mezclar con refactor de base.
- Objetivo: aligerar sistema y soporte.

PASO 10: Hardening de produccion
- Alcance: backups, rollback, monitoreo, alertas, checklist operativa final.
- Objetivo: salida a produccion con riesgo controlado.

PLANTILLA DE CHECKLIST POR PASO
- [ ] Alcance y no-alcance definidos
- [ ] Contratos estables documentados
- [ ] Cambios limitados al dominio
- [ ] Smoke tests ejecutados
- [ ] Evidencia de resultados
- [ ] Riesgos identificados
- [ ] Rollback probado o documentado

RESULTADO ESPERADO
Con este camino, cada mejora se implementa como bloque aislado.
Si algo falla, se revierte ese bloque sin comprometer el resto del sistema.
